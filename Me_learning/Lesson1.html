Lesson 1: The Blueprint (
network.yaml
)

This file is like a blueprint for a house. We don't build the house ourselves; we just give this blueprint to AWS (the construction crew) and they build it exactly as described.

This file is written in YAML, which is just a list of instructions.

The Major Parts:

  1. Parameters (The Inputs)

    • Think of these like questions AWS asks you before building:
    • "How big do you want the network?" (That's the 
      VpcCidr
      ).
    • We gave it a default answer: 
      10.0.0.0/16
       (think of this as the size of your land).

  2. Resources (The Things to Build)

    • The VPC (
      MyVPC
      )      : This is your private plot of land in the AWS cloud. No one else can enter unless we let them.
    • Internet Gateway (
      MyIGW
      )      : This is the front door. Without it, your private land is completely cut off from the internet.
    • Public Subnet (
      PublicSubnet
      )      : This is a specific "room" in your house. We call it "Public" because we allow people from the internet to visit it (like a guest room).
    • Route Table (
      PublicRouteTable
      )      : This is a map. It tells traffic inside your house: "If you want to go to the internet (0.0.0.0/0), go through the Front Door (Internet Gateway)."

Summary of what this file does: It creates a private space (VPC), adds a front door (Gateway), builds a room (Subnet), and hangs a map on the wall (Route Table) so traffic knows how to get in and out.

Ready for Lesson 2 (The Website)?


That's a great observation! You've spotted the "magic" that makes these templates dynamic.

1. Definitions & References (
!Ref
)

Think of this like using variables in algebra or a nickname.

  • Definition: When we create the VPC, we give it a logical name: 
    MyVPC
    .
    yaml
    MyVPC:  <-- This is the nickname we define
      Type: AWS::EC2::VPC
  • Reference (
    !Ref
    )    : Later, when we make the Internet Gateway, we need to attach it to that specific VPC. Instead of typing a long ID that we don't know yet, we just say "Use 
    MyVPC
    ".
    yaml
    VpcId: !Ref MyVPC  <-- "AWS, please plug this into the thing I named MyVPC above"
    AWS figures out the actual IDs strictly at the moment it builds them. It builds the VPC first, gets the ID, and then passes it to the Gateway.

2. Where do these templates come from?

There are three main ways people get them:

  1. Documentation (The Cookbook): AWS has a massive manual called the "CloudFormation User Guide". It lists every resource (like 
    AWS::EC2::VPC
    ) and all its properties. We look up the ingredients we need and type them out.
  2. Quick Starts (Frozen Meals): AWS provides pre-made templates for common things (like "WordPress site" or "Standard Network"). You can download these and just tweak them.
  3. Writing from Scratch (Home Cooking): This is what we did here. I acted as the chef who knows the recipes (the syntax) and wrote a custom meal plan just for you, keeping it simple without the extra "fluff" that generic templates often have.

Lesson 2: The House (
website.yaml
)

Now that we have the land (network), we need a house to live in. In the cloud, a "Static Website" lives in a Bucket (S3).

This template is simpler but has two very cool new concepts.

1. Resources (The Construction)

  • The Bucket (
    WebsiteBucket
    )    : This is the hard drive in the sky where we put your 
    index.html
    .
    • Public Access: Notice we explicitly set "Block Public Access" to 
      false
      . Normally, S3 buckets are private (like a safe). But since this is a website, we have to unlock the safe so the world can see it.
  • The Policy (
    WebsiteBucketPolicy
    )    : This is the rule book taped to the front of the safe.
    • It says: *"Allow (Effect: Allow) Everyone (**Principal: **) to Read (Action: GetObject) anything inside this bucket."

2. Conditions (The Logic)

This template is "smart".

  • Conditions
     block    : It defined logic called 
    HasBucketName
    .
  • How it works:
    • If you gave it a specific name (like "egirgis-lab"), it uses it.
    • If you didn't, it tells AWS: "You pick a random name for me."
    • It does this with 
      !If
       logic inside the resource: 
      !If [ HasBucketName, !Ref BucketName, !Ref AWS::NoValue ]

3. Outputs (The Return Gift)

At the very bottom, it has an Outputs section.

  • After building the house, it doesn't just say "Done."
  • It calculates and gives us back the WebsiteURL.
  • This is how our Python script (which we'll see next) knows what to print out for you at the end: 
    SUCCESS: Website is live at: ...

Lesson 3: The Construction Crew (
aws_lib/stacks.py
)

Blueprints (.yaml files) are useless if you don't give them to a builder. This Python file is that builder.

It doesn't "know" how to build a VPC itself. Its job is to talk to the AWS CloudFormation service (the Master Builder at AWS).

The Main Jobs:

  1. deploy()
     - "Make it happen"

    • This function reads your blueprint file (
      network.yaml
      ).
    • It checks: "Does this stack already exist?"
    • If No: It says "AWS, please Create this Stack."
    • If Yes: It says "AWS, please Update this Stack with any changes."
    • Key Point: It handles the "Diff" logic. If you change one line in the YAML, this function helps AWS figure out just that one change.

  2. _wait_for_completion()
     - "Are we there yet?"

    • Sending the blueprint happens instantly (milliseconds).
    • But building a VPC takes ~2 minutes.
    • This code puts the script to sleep and asks AWS every few seconds: "Is it done yet? ... Is it done yet?"
    • This is why your terminal pauses until everything is green and ready.

  3. destroy()
     - "Tear it down"

    • This simply tells AWS: "Delete this stack ID."
    • AWS cleans up everything associated with that blueprint automatically.

Summary

  • YAML files = The Instructions (What to build).
  • Python script = The Messenger (Giving instructions to AWS and waiting for the result).


1. 
cli.py
 (The Commander)

  • Role: This is the "Boss". It's the only file you ever run directly (
    python cli.py ...
    ).
  • Job: It reads what you type in the terminal.
    • If you type 
      deploy
      , it wakes up 
      stacks.py
       and says "Go build the house."
    • If you type 
      audit
      , it wakes up 
      audit.py
       (see below) and says "Go check the bills."
  • Why it matters: It keeps everything organized so you don't have to remember 5 different script names. You just remember 
    cli.py
    .

2. 
aws_lib/core.py
 (The Bouncer / Key Master)

  • Role: Authentication.
  • Job: BEFORE 
    cli.py
     can order 
    stacks.py
     to build anything, it needs permission.
    • core.py
       looks for your keys (MFA/Profile).
    • It creates a "Session" (a secure temporary badge).
    • It hands this badge to everyone else (
      stacks.py
       and 
      audit.py
      ) so they can pass through the AWS gates.

3. 
aws_lib/audit.py
 (The Inspector)

  • Role: Verification & Cost.
  • Job: It's a sibling to 
    stacks.py
    , but it doesn't build anything.
    • It uses 
      boto3
       (the AWS phone line) to ask: "How much money did we spend?" or "List all running servers."
    • It helps you double-check that your code (
      stacks.py
      ) didn't leave expensive servers running by accident.

So, the full flow is:

  1. You run 
    cli.py
     ->
  2. cli.py
     asks 
    core.py
     for a Badge ->
  3. cli.py
     gives the Badge to 
    stacks.py
     ->
  4. stacks.py
     reads 
    network.yaml
     and builds the VPC.
